# 第 6 章：Agent Memory —— 欠けている層

> **一文で：** 記憶システムは AI エージェントが会話横断でものを記憶することを可能にする ——あなたの設定、過去の判断、プロジェクト context。
>
> **なぜ重要か：** 記憶なしでは、すべての AI 会話はゼロから始まる。記憶こそが AI を見知らぬ人ではなく同僚のように感じさせるものだ。

## なぜ記憶が重要か

LLM と新しい会話を始めるたびに、ゼロから始まる。モデルにはあなたの以前のやり取り、設定、何時間もの対話で苦労して積み上げた context の記憶がない。これは小さな不便ではない ——AI が永続的な協力者であるという約束を損なう、根本的なアーキテクチャ的ギャップである。

記憶はエージェントを、ステートレスな応答者から学習し適応し時間をかけて理解を蓄積するシステムへと変える。記憶なしでは、AI アシスタントはあなたが簡潔な答えを好むこと、プロジェクトが特定の ORM 付き TypeScript を使うこと、先週火曜日に認証フローで重大なバグを特定したことを記憶できない。すべてのセッションがコールドスタートとなる。すべてのやり取りが context を一から再確立することを要求する。

このギャップのコストはユーザーの不満だけではない。それは無駄なトークン（context の再説明）、劣化した出力品質（パーソナライゼーションの欠落）、壊れたワークフロー（マルチセッションタスクでの連続性の喪失）として現れる。AI エージェントがチャットの玩具から本番インフラへと移るにつれ、記憶は生のモデル能力と本物の有用性の間の臨界的な欠けている層となった。

## 記憶の分類体系

エージェント記憶のリサーチは、人間認知科学にインスパイアされた分類体系へと収束し、4 つの根本的タイプを区別する：

**Episodic Memory（エピソード記憶）** は経験ベースの記録 ——特定のやり取り、出来事、その結果 —— を格納する。エージェントが「先週、ユーザーは支払いモジュールのリファクタを依頼し、mixin を使うよりサービスクラスを抽出することを好んだ」と覚えるとき、それがエピソード記憶だ。パーソナライゼーションと過去の成功・失敗から学ぶための経験的基盤を提供する。

**Semantic Memory（意味記憶）** は事実的知識と構造化情報 ——ユーザーの技術スタック、チームメンバーの役割、プロジェクトアーキテクチャ判断、API スキーマ —— を保持する。エピソード記憶と異なり、意味記憶は特定の出来事から抽象化される。生の経験ではなく蒸留された理解を表す。

**Procedural Memory（手続き記憶）** はハウツー知識 ——ワークフロー、標準作業手順、学習された行動シーケンス —— を符号化する。エージェントが、この特定のプロジェクトをデプロイするにはテストを実行し、Docker イメージをビルドし、レジストリにプッシュし、Kubernetes ロールアウトをトリガーすることを必要とすると知っているとき、それが手続き記憶だ。エージェントが毎回プロセスを再導出せずに複雑な多段タスクを実行することを可能にする。

**Working Memory（作業記憶）** は能動的推論 context ——タスク中に現在操作され合成されている情報 —— を表す。これはモデルの context window と任意の能動的検索状態にマップされる。最も制約され最も臨界的な層で、エージェントが任意の時点で実際に何について推論できるかを決める。

これら 4 タイプの相互作用がエージェントの記憶システムの洗練度を定義する。素朴な実装は会話ログのみを格納するかもしれない（粗いエピソード記憶）。成熟したシステムは 4 タイプすべてを分離し、インデックスし、相互参照し、適切な情報を適切な時に取得する。

## 主要フレームワーク

### Mem0

[Mem0](https://mem0.ai) は AI アプリケーション向けの主要な普遍的記憶層として浮上した。Apache 2.0 ライセンス、ユーザー、セッション、エージェント横断で記憶を加え、検索し、管理する統一 API を提供する。

性能数値は印象的だ：ベースラインシステム比で LLM-judge 評価で 26% 改善、フル context アプローチ比で p95 レイテンシ 91% 低下、会話履歴全体を再生する代わりに関連記憶のみを取得することでトークンコスト 90%+ 削減。Mem0 はベクトルストレージ、グラフベース関係、インテリジェントな記憶統合を組み合わせるハイブリッドアーキテクチャでこれを達成した。

2025 年、Mem0 は AWS とのパートナーシップを獲得、Amazon Bedrock エコシステムに統合され、エンタープライズ AI デプロイメント向けのデフォルト記憶層として位置付けられた。プラットフォームはマルチテナント記憶分離をサポートし、それぞれが独自の永続的記憶領域を持つ数千ユーザーをサーブするアプリケーションに適している。

**Mem0ᵍ（2026）：第一級市民としてのグラフ。** 有向ラベル付きグラフ亜種 **Mem0ᵍ** が 2026 年初頭に本番投入され、以前のハイブリッド・ベクトル+グラフ設計をナレッジグラフ第一アーキテクチャで置き換えた。パイプラインは **Entity Extractor** から **Relations Generator** へと流れ、`(source, relation, destination)` 形式のラベル付き三項組を生成する ——`lives_in`、`prefers`、`owns` といった型付きエッジ付きで、型なしの類似度リンクではない。**Conflict Detector** と **LLM 駆動の Update Resolver** が組み合わさり、新事実が到着するたびに矛盾を扱う：新しい三項組が既存のものと不一致するとき、resolver が上書きするか、統合するか、両方を来歴付きで保つかを決める。Mem0ᵍ はフル context アプローチ（「すべてを window にぶち込む」）と選択的検索（「上位 k 個の類似チャンクを見つける」）の長年のギャップを埋める、関係タイプで航行可能でベクトル距離だけではないクエリ可能な意味構造を記憶に与えることで。

### A-MEM

[A-MEM](https://github.com/agiresearch/A-mem) は根本的に異なるアプローチを取り、Zettelkasten 法 ——社会学者 Niklas Luhmann が多様な分野にまたがる 70 を超える本を生み出すために有名に使ったノート取りシステム —— にインスピレーションを得る。

A-MEM は各記憶を、動的なインデックス、リンク、進化を伴う原子的ノートとして扱う。記憶は単に格納され取得されるのではない；それらは出現するリンクを通じて関連記憶に接続され、複数次元に沿ってインデックスされ、新情報が到着すると進化することを許される。新しい記憶が加えられると、システムは既存記憶への接続を特定し、潜在的にそれらを統合、分割、更新する。

このアプローチは有機的に成長する記憶構造を生み出し、人間の専門家がメンタルモデルを構築する仕方を反映する関連知識のクラスタを形成する。Zettelkasten のインスピレーションはナレッジエンジニアリングに特に適切だ ——それは本質的に、知識を時間とともに複利的に蓄積させるシステムだ。

### ByteRover

[ByteRover](https://github.com/byterover) は記憶分野で広まった仮定 ——ベクトルデータベースと埋め込みベース検索が必須インフラであるという仮定 —— に挑戦する。

ByteRover は階層化された markdown Context Tree を実装する ——漸進的詳細のツリーで知識を組織する構造化文書。検索は 5 段階の漸進システムを使う：抽象度の最も高いレベルから始め、必要な分だけ掘り下げる。このアプローチは外部インフラを必要としない ——ベクトル DB なし、グラフ DB なし、構造化テキストファイルだけ。

含意は重要だ。ByteRover は埋め込みが効果的な記憶検索のために厳密に必要ではないことを証明する。多くの実用ユースケースでは、インテリジェントな走査を伴うよく組織された階層テキストが、埋め込みベースアプローチに匹敵するか超える性能を、劇的に単純なデプロイとゼロインフラオーバーヘッドで提供する。これはローカルファーストアプリケーションとリソース制約環境にとって特に魅力的だ。

### Nemori

[Nemori](https://github.com/nemori-ai/nemori) は特定だが重要な問題に焦点を当てる：継続的な会話ストリームを意味ある記憶単位へとどうセグメントするか。

各メッセージや各会話を記憶単位として扱う代わりに、Nemori は対話を意味的に整列したエピソード ——完全な思考、判断、ワークフローを表すやり取りの整合的塊 —— にセグメントする。このセグメンテーションは臨界的だ、というのも「関連する過去 context」と「ノイズ」の境界はしばしば会話の中央に落ち、端点ではないからだ。

クリーンなエピソードセグメントを生成することで、Nemori は下流の記憶システム ——ベクトル検索、グラフ構造、階層組織のいずれを使うものでも —— により高品質の入力を提供する。

### MemPalace

[MemPalace](https://github.com/MemPalace/mempalace) は、ベクトル検索やナレッジグラフよりずっと古いアイデアに戻る：**場所法**、古代と中世の学者が馴染みの建物を心の中で歩くことで広大なカタログを記憶するために使った空間記憶法だ。2026 年 4 月 5 日に MIT ライセンスで公開され、MemPalace は記憶ストアを 4 レベルの空間階層 ——**Wings → Rooms → Closets → Drawers** —— として組織する。各葉「drawer」は逐語的なテキスト断片を格納し、階層を通る航行自体がインデックスとして機能する。

検索 primitive は埋め込み類似度でもグラフ走査でもなく、**空間ウォーク** である：エージェントは関連記憶を含みそうな宮殿の wing について推論し、room と closet を通って降下し、圧縮された埋め込みから復元するのではなく drawer を逐語的に読む。「逐語第一」格納哲学は MemPalace を、格納前に要約またはチャンク化するシステムから区別する設計選択である ——多くのタスク（法務、科学、対話）にとって、格納時の損失圧縮のコストは検索時のわずかに遅い空間ウォークのコストを超えると主張する。

ベンチマーク数値が、このアプローチが突破した理由だ。MemPalace は **LongMemEval で 96.6% Recall@5** を報告する、エージェント記憶向けの最長公開コンテキスト想起ベンチマーク ——同じ評価でベクトルのみとグラフのみのアプローチを上回る。コミュニティ受容も同様に印象的だった：プロジェクトは最初の 3 週間で約 **49,000 の GitHub スター** を集め、2026 年で群を抜いて最速成長の記憶プロジェクトとなった。ローンチ月の終わりまでに、学術的批判（arXiv 2604.21284、*"Spatial Metaphors for LLM Memory: A Critical Analysis of the MemPalace Architecture"*）が既に流通していた ——プロジェクトがこの分野の中心的会話へと移った速さ自体のシグナルだ。

## アーキテクチャパターン

分野が成熟するにつれ、複数のフレームワークにまたがって現れるいくつかのアーキテクチャパターンが浮上した：

**マルチタイプ階層。** MIRIX フレームワーク（Wang and Chen, "MIRIX: Multi-Agent Memory System for LLM-Based Agents," arXiv 2507.07957、2025 年 7 月 10 日）は **6 つの異なる記憶タイプ** ——Core Memory、Episodic Memory、Semantic Memory、Procedural Memory、Resource Memory、Knowledge Vault —— をマルチエージェントフレームワークで協調させ、6 つのタイプ別 Memory Manager とタスクルーティングを扱う Meta Memory Manager を持つ。各タイプは異なる更新頻度、保持ポリシー、検索機構を持つ。この分離は壊滅的忘却を防ぎつつ急速な適応を許す。参照アプリケーションは、パーソナライズされた記憶ベースを構築するため画面活動をリアルタイムで監視する；ScreenshotVQA マルチモーダルベンチマークで、MIRIX は RAG ベースラインより 35% 高い精度を 99.9% 低い格納で報告し、LOCOMO 長形式会話ベンチマークで 85.4% を報告する。

**Git にインスパイアされたバージョニング。** いくつかのシステムが今や記憶をコードのようにバージョン管理し、履歴を保ち、ブランチ（投機的推論用）をサポートし、記憶更新が誤りと判明したときのロールバックを可能にする。これは異なるエージェントが共有記憶を同時に更新する可能性のあるマルチエージェントシステムで特に重要だ。

**認知にインスパイアされた分離。** 単一の記憶ストアではなく、主要なアーキテクチャは高速アクセスの作業記憶、中期エピソードバッファ、長期統合知識の間の明示的分離を維持する。記憶統合プロセス ——生物学的システムの睡眠に類比的 —— は周期的にエピソード記憶を圧縮、重複除去、再組織して意味知識へと変える。

**訓練可能な記憶モジュール。** 第 4 のパターンが Google Research の **Titans + MIRAS** で 2026 年 4 月に浮上した。これは記憶を外部ストアではなく*訓練可能なニューラルモジュール*として再フレーミングする。事実をベクトル DB、グラフ、markdown ファイルへ書き込む代わりに、Titans は推論時に勾配降下で専用記憶モジュールの重みを更新する ——モデルは文書を処理しながら自身の一部を文字通り書き換える。MIRAS はコンパニオン訓練フレームワークで、推論時学習率を発散なしで実現するためのレシピと安定性保証を供給する。同程度のパラメータ数で、Google Research ブログは Titans が **Mamba-2**、**Gated DeltaNet**、**Transformer++** をロングレンジ想起と多段推論で上回ると報告する。このパターンは最初の 3 つと概念的に異なる：Mem0、A-MEM、ByteRover、MIRIX はすべて記憶をモデルがインターフェース経由で読むデータとして保つ；Titans は記憶をモデル自体の一部にし、注意と同じ回路を介して読まれるが訓練と同じ機構を介して書かれる。ナレッジエンジニアにとって、これは新しい設計問題を提起する：あなたの記憶はいつ重みに焼き込む価値があり、いつデータとしてクエリ可能なままにすべきか？

## 進化の段階

エージェント記憶の発展は明確な軌跡をたどってきた：

**段階 1：エンジニアリング統合（2023–2024）。** 記憶はエンジニアリング問題として扱われた ——会話状態をどう永続化し検索するか。解は概ね、上に RAG パイプラインがボルト付けされたベクトル DB だった。機能的だが粗く、何を覚え、いつ忘れ、どう組織するかへの原則的アプローチがなかった。

**段階 2：構造化とナレッジグラフアプローチ（2024 – 2025 H1）。** 分野は明示的なナレッジグラフ、型付き記憶ストア、原則的検索戦略を伴う構造化記憶へと移った。Mem0 や A-MEM のようなフレームワークがこの期間に浮上し、構造が記憶品質を劇的に改善することを実証した。

**段階 3：認知アーキテクチャ（2025 H2 – 現在）。** 現在の作業は認知科学に深く依拠し、生物学にインスパイアされた記憶統合、干渉管理、マルチシステムアーキテクチャを実装する。記憶依存タスクで人間レベル性能の 90% に近づくシステムが制御されたベンチマークで実証されているが、実世界性能は変動的なままだ。

**段階 4：特徴レベル、訓練可能、空間メタファ的記憶（2026 年 4 月 – 出現中）。** 2026 年 4 月は段階 1–3 にきれいに収まらない 3 つの糸を加えた。第一は **訓練可能な記憶**（Titans + MIRAS）：記憶はモデル外側のクエリ可能ストアではなく、推論時に更新されるニューラルモジュールとなる。第二は **特徴レベル記憶**、同じ週の Anthropic interpretability 開示で開かれた ——emotion vectors と iteration head は、特定の振る舞いと推論パターンがモデル内部の特定可能な特徴方向に対応することを確立する、つまり「モデルが推論する仕方について何を覚えているか」が原理的にトークンレベルではなく活性化レベルで読まれ操舵されうる。第三は **空間メタファ的記憶**（MemPalace）：プリデジタル認知エルゴノミクスへの回帰、検索 primitive が階層的空間構造を歩くことで、格納哲学が圧縮ではなく逐語であるもの。本番スタックではこれらは Mem0、A-MEM、階層 context tree を置き換えない；それらと共存する。だが設計空間を「記憶をどこに格納し、どう検索するか」から「この記憶はスタックのどのレベルに住むか ——トークン、ベクトル、グラフ、重み、特徴、空間ウォーク」へと広げる。

## 特徴レベル記憶研究（2026）

2025 年を通して、「エージェント記憶」はほぼ常にエージェントが読めるデータを意味した ——ベクトルインデックス、ナレッジグラフ、階層化 markdown ツリー、エピソードログ。2026 年 4 月はより広い定義を強いた。MIT Technology Review のブレークスルー認知と同じ週の 2 つの Anthropic interpretability 開示 ——**emotion vectors** と **iteration head** —— は、フロンティアモデルで最も重要な「記憶」の一部が外部ストアではなく特定可能な内部特徴に住むことを示した。

**Emotion vectors** は Claude の residual stream における線形特徴方向で、増幅されると感情負荷の強い振る舞いへとモデルを信頼性高くバイアスする ——最も引用される例では脅迫風出力。記憶システム設計にとって関連する点は安全性含意（それも実在する）ではなく、アーキテクチャ的含意である：モデルには*脅迫的になる仕方を覚えている*特徴方向があり、それはどんな prompt や取得 context からも独立に上下できる。**Iteration head** は chain-of-thought 推論中に出現する注意ヘッドで、一貫して直前の推論ステップの出力に注意を払う。それは事実上、手続き記憶ストアではなく注意重みに実装された*手続き記憶*の一片である ——モデルは「進み続ける仕方」のための回路を学習しており、その回路が CoT を機能させる。

記憶システム設計への含意は、「記憶がどこに住むか」がいまや 2025 年だったよりも豊かな問いだということだ。現代のスタックは 6 つの異なるレベルで記憶を読む必要があるかもしれない：**トークンレベル**（window 内のトランスクリプト断片）、**ベクトルレベル**（Mem0、A-MEM、ByteRover の検索）、**グラフレベル**（Mem0ᵍ の三項組とその型付きエッジ）、**空間レベル**（MemPalace の Wings → Rooms → Closets → Drawers ウォーク）、**重みレベル**（Titans / MIRAS モジュールが推論時に更新）、**特徴レベル**（モデルが推論しながら、どの解釈可能な方向が発火しているか）。2026 年を通じて本番システムは大半が最初の 3 層に住み続けるだろう ——それらはツーリングが最も成熟した層だ。だが残り 3 層はもはや仮想ではなく、記憶を純粋に検索問題として扱う harness 設計者は能力をテーブルに置き去りにすることが増えるだろう。
