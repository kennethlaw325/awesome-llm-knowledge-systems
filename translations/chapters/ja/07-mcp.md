# 第 7 章：MCP —— 勝利した標準

> **一文で：** MCP は、AI が単一のプロトコルを通じてあらゆる外部ツールやデータソースに接続することを可能にする普遍的標準だ。
>
> **なぜ重要か：** MCP は AI のための USB のようなもの ——その前は、各ツールが独自のカスタム接続を必要とした。今や、どこでも動く 1 つのプラグがある。

## MCP とは何か

Model Context Protocol（MCP）は Anthropic が作ったオープンプロトコルで、AI モデルが外部ツール、データソース、サービスへとどう接続するかを標準化する。MCP 以前、すべての AI 統合はオーダーメイドだった ——カスタム関数定義、独自ツール形式、手作業の認証、脆い糊コード。MCP はこの断片化を普遍的インターフェースで置き換えた：どのモデルもツールを発見し、認証し、呼び出すために使える単一プロトコル。

MCP を AI のための USB と考えよ。USB 以前、すべての周辺機器は独自のコネクタ、ドライバ、プロトコルを必要とした。USB は周辺機器をより強力にしたのではない ——相互運用可能にしたのだ。MCP は AI ツールエコシステムに対して同じことをする。単一の MCP サーバー実装が、Claude、GPT、Gemini、または他のプロトコルを話すモデルにサーブできる。単一の MCP クライアントが、カスタム統合コードなしで数千のサーバーに接続できる。

## タイムライン：ローンチから Linux Foundation へ

**2024 年 11 月。** Anthropic が MCP をオープンソース仕様として、Python と TypeScript の参照実装と並んで公開する。初版は少数の参照サーバー（filesystem、GitHub、Slack）とカスタムサーバー構築のための SDK サポートを含む。受容は慎重に楽観的だ ——AI コミュニティは以前にも相互運用性標準が提案されるのを見てきた。

**2025 年 4 月。** ローンチ 6 か月後、MCP は累計 SDK ダウンロード 800 万を超え、コミュニティは 5,800 を超えるサーバーを構築している。エコシステムは誰の予期よりも速く成長する。臨界的に、競合 AI ラボが代替を作るのではなくプロトコルを採用し始める。

**2025 年後半。** MCP は Linux Foundation 傘下の新エンティティ Agentic AI Foundation に寄贈され、Anthropic、Block、OpenAI からの設立ガバナンスを得る。この動きは「Anthropic が標準を支配する」懸念を中立化し、ベンダー固有プロトコルではなく真の業界標準として MCP を確立する。

**2026 年（現在）。** MCP SDK ダウンロードは月間で全パッケージレジストリ合計 9,700 万を超える。エコシステムは Foundation が保守する 75+ の公式コネクタを含み、コミュニティ構築のサーバーは数千多い。プロトコルは AI アプリケーションのデフォルト統合層となった、REST がウェブ API のデフォルトとなったのと同様に。

## なぜ勝ったか

MCP の急速な支配を説明するいくつかの要因：

**最初からオープンスタンダード。** MCP は完全な仕様付きで寛容なライセンスのもとで公開された、SDK だけではなく。これは競合者が Anthropic のコードや好意に依存せずに実装できることを意味した。

**シンプルなアーキテクチャ。** プロトコルには 3 つのコアコンセプトがある：クライアント（AI モデル側）、サーバー（ツール / データ側）、トランスポート（どう通信するか）。サーバーは JSON Schema 記述付きでツールを露出する。クライアントはそれらを発見し呼び出す。トランスポートは通信層を扱う（stdio、SSE 付き HTTP、WebSocket）。このシンプルさが採用を些細にした ——有能な開発者は 1 時間以内に MCP サーバーを構築できる。

**ベンダー中立ガバナンス。** OpenAI を共同統治メンバーとする Linux Foundation への寄贈が、最後の信頼できる反対意見を消し去った。主要競合者が標準を共同統治するとき、その標準は本物の中立だ。

**ネットワーク効果。** すべての新 MCP サーバーは、すべてのクライアントにとってプロトコルをより価値あるものにし、逆もまた然り。エコシステムが有用なサーバーの臨界質量を超えると、MCP をサポートしないコストが採用するコストより高くなった。

## 鍵となる採用者

MCP 採用の幅広さがその支配の物語を語る：

- **OpenAI** は ChatGPT プラットフォームと Agents SDK に MCP サポートを統合し、GPT モデルがどの MCP サーバーも使えるようにした。
- **Google DeepMind** は Gemini に MCP クライアントサポートを加えた、エコシステム互換性を主要動機として引用した。
- **Microsoft** は最も遠くまで行った、Windows OS レベルに MCP サポートを構築し、Azure AI services、Copilot、GitHub Copilot 全体に統合した。
- **Amazon / AWS** は 2026 年 4 月に **Bedrock AgentCore Runtime** を、最初の本番双方向 MCP ランタイムとして出荷し、標準プロトコルの上に elicitation と sampling を追加した（下の「ステートフル MCP への遷移」を参照）。
- **GitHub** は Copilot 拡張機能システムを MCP の上に再構築し、独自ツール形式を置き換えた。
- **Figma、Slack、Block、Notion** はすべて公式 MCP サーバーを出荷し、プラットフォームをどの AI ワークフローでも第一級市民にする。
- **Hugging Face** はモデルハブと並んでコミュニティ MCP サーバーのレジストリをホストする。
- **LangChain** その他の編成フレームワークはネイティブ MCP クライアントサポートを提供し、デフォルトのツール統合方法とする。

**合成次元としての MCP。** より微妙な 2026 年の発展：MCP ツール選択は固定された基盤としてではなく、自動化された harness 合成システムの*変数*の 1 つとして現れ始めている。AgentFlow 論文（arXiv 2604.20801、2026 年 4 月）はエージェント役割、prompt、*ツール*、通信トポロジ、協調プロトコルを、合成ループが探索する型付きグラフ DSL の 5 次元として扱う。MCP 認識の harness 設計者にとって、これは両刃だ。MCP の標準化はツール次元を真に探索可能にする ——合成ループは統合層を書き直さずに 1 つの MCP サーバーを別のものに交換できる。だが同じ標準化は MCP サーバー自体を選択候補にする：問いはもはや「どのサーバーをインストールするか」ではなく「観測可能性圧力下で、私の harness は*どのサーバーを使うように進化するか*」である。

## ナレッジエンジニアリングへの接続

MCP それ自体はナレッジエンジニアリング技法ではない ——ナレッジエンジニアリングをスケーラブルにするインフラ層だ。本レポートで議論されたすべてのアプローチ ——RAG パイプライン、ナレッジグラフ、記憶システム、context engineering パターン —— は外部知識をモデルに供給することを必要とする。MCP はその供給がどう起こるかを標準化する。

実用例を考えよ：会社 wiki にアクセスし、データベースをクエリし、ベクトルストアを検索し、ファイルシステムから読む必要のあるナレッジ強化 AI エージェント。MCP なしでは、これらの統合の各々がカスタムコード、カスタム認証、カスタムエラーハンドリングを必要とする。MCP では、各々が MCP サーバーで、エージェントは均一なインターフェースを通じてそれらすべてに接続する。

これが MCP がナレッジエンジニアリングエコシステムの中心である理由だ。RAG や記憶を置き換えるのではない ——それらをモデルとプラットフォーム横断で構成可能、ポータブル、相互運用可能にする配管を提供する。

## Meta-Tool パターン

MCP 採用から浮上した最も重要なアーキテクチャパターンの 1 つが、Meta-Tool パターン、ツール用 progressive disclosure とも呼ばれるものだ。

問題：MCP エコシステムが拡大するにつれ、エージェントは数十か数百のツールにアクセスできるようになるかもしれない。すべてのスキーマを context window にロードするのは無駄だ ——大半のツールは任意のタスクに無関係。29 のツールスキーマがロードされていれば、モデルが決して使わないツール記述に数千トークンを焼き、モデルが間違ったツールを選ぶ確率を増やす。

解：すべてのツールスキーマを最初にロードする代わりに、2 つのメタツールをロードする：エージェントが説明で関連ツールを検索できる **discovery tool** と、参照で発見されたツールを呼び出す **execution tool**。エージェントは最初に現在のタスクに利用可能なツールを発見し、必要なスキーマだけをロードする。

このパターンはツール重い環境で context window 消費を 80–90% 削減し、判断空間を減らすことでツール選択精度を改善する。10–15 を超える利用可能ツールを持つどんなデプロイメントにとってもベストプラクティスとなった。

## 2026 年の課題

成功にもかかわらず、MCP は重要な未解決課題に直面する：

**監査トレイル。** エンタープライズ環境では、すべてのツール呼び出しがロギング、帰属、コンプライアンス追跡を必要とする。現在の MCP 仕様は監査トレイル形式を規定しておらず、実装間の不整合を招いている。

**SSO 統合認証。** MCP の認証ストーリーはローンチ以来著しく改善したが、エンタープライズ SSO システム（SAML、OIDC）と完全なクライアント・サーバー・トランスポートチェーン全体で統合することは依然として摩擦が重い。最近批准された MCP 用 OAuth 2.1 フローはこの一部に対処するが、採用は依然として展開中だ。

**ゲートウェイの振る舞い。** 組織が MCP をスケールでデプロイするにつれ、レート制限、アクセス制御、リクエストルーティング、監視のためのゲートウェイ層を必要とする。エコシステムには標準 MCP ゲートウェイ仕様がなく、独自解を招いている。

**Config ポータビリティ。** MCP サーバー設定（どのサーバーに、どんな認証情報で、どの順番で接続するか）は標準化されていない。MCP セットアップをマシン、チーム、組織間で移すには手動再設定を必要とする。

**ツール記述品質。** MCP の有効性はツール記述の品質に完全に依存する。貧弱に記述されたツールは不正な呼び出しを招く。ツール記述品質の標準もリンタもなく、認証もない。

**スキーマバージョニング。** MCP サーバーがツールスキーマを更新するとき、既存のクライアントが壊れる可能性がある。プロトコルには組み込みのバージョニングと互換性ネゴシエーション機構がない。

**ツールの重複。** 数千のコミュニティサーバーで、複数のサーバーがしばしば重複機能を提供する。プロトコルには重複除去や選好表明の機構がない。

## 2026：ステートフル MCP への遷移

最初の 18 か月を通じて、MCP は根本的に **ステートレス RPC プロトコル** だった：クライアントがサーバーを呼び、サーバーが結果を返し、会話は前進する。このモデルは単純なツール呼び出し（ファイルを読む、API をクエリする、メッセージを投稿する）にはよく機能したが、長時間動くものや対話的なものには苦労した。2026 年 4 月、2 つの発展が MCP を真のオーケストレーション層へと押し始めた。

**SEP-1686：Tasks Primitive。** 2026 年 4 月の MCP Dev Summit で導入された **SEP-1686** は、プロトコルに永続的タスク状態機械を追加する。各長時間ツール呼び出しは、永続的 ID と 5 つの状態のいずれか（`working`、`input_required`、`completed`、`failed`、`cancelled`）を持つ **task** となる。クライアントはタスクを開始し、切断し、後で再接続して状態をチェックしたり結果を取得したりできる ——**call-now / fetch-later** パターン。タスク ID はサーバー送信通知と相関され、クライアントはオープン接続を保たずに進捗イベントをサブスクライブできる。Primitive は MCP が以前オーダーメイドの回避策へと押し込んでいたワークフローに真っ直ぐ向けられている：

- 完了に分や時間を要する **データパイプライン**
- 大規模リポジトリにまたがる **コード移行**
- 完全 CI スイートのための **テスト実行**
- サブエージェントを派生させ外部 API を待つ **ディープリサーチ**
- 1 つのエージェントが別のエージェントに非同期に作業を渡す **マルチエージェントシステム**

SEP-1686 は 2026 年 4 月 MCP 仕様で実験的として出荷される。これはステートレス RPC から永続的、再開可能なオーケストレーションへの最初の真剣な一歩だ。

**Amazon Bedrock AgentCore：最初の双方向ランタイム。** 同じく 2026 年 4 月、AWS は **Bedrock AgentCore Runtime** を出荷、**双方向**通信を実装する最初の本番 MCP ランタイムである。標準 MCP はクライアント起点だ：クライアントがツールを呼び、サーバーが応答する。AgentCore は 2 つのサーバー起点機能を加える：

- **Elicitation。** 実行中、サーバーは一時停止し JSON スキーマに対してユーザーから構造化入力を要求できる。これは以前カスタム UI 層を必要とした多段ワークフローを、プロトコルレベルの相互作用に変える。移行ツールは止まって「どのブランチを対象にすべきか？」と型付き応答付きで尋ねられる。
- **Sampling。** サーバーは自身のモデル認証情報を保持せずに、クライアントから LLM 生成補完を要求できる。サーバーは prompt とサンプリングパラメータを提供する；クライアントは自身のモデルを動かして補完を返す。これにより、ツールサーバーが LLM 駆動の作業（要約、分類、抽出）をする一方で、ユーザーのモデル、クォータ、課金はクライアント側に留まる。

両者を併せて、elicitation と sampling は MCP の双方向プロトコルサーフェスを完成させる。サーバーは呼び出しに応答するだけでなく、会話の一部を**駆動**できる。

**含意。** 2025 年の MCP のフレーミングは「ステートレスツールのための均一なプラグ」だった。2026 年のフレーミングはより広い：MCP は **ステートフル、対話的、双方向のエージェントオーケストレーション** のためのプロトコル層になりつつある。ツールサーバーはもはや受動的ではない ——永続的状態を保持し、ユーザー入力のために停止し、クライアントから LLM 作業を要求できる。ナレッジエンジニアリングにとって、これは長時間検索パイプライン、段階的データ強化、人間ループ内ワークフローがついにネイティブに動くプロトコルを得たことを意味する。

## 市場の文脈

業界アナリストは AI ツール統合市場 ——MCP がその支配的プロトコル —— を 2025 年に 18 億ドルと予測しており、エンタープライズ採用が加速するにつれ著しい成長が予期される。この数字は MCP インフラ、サーバー開発、ゲートウェイサービス、関連ツーリングを包含する。
